# 📊 Управління журналом аудиту

> Версія: 1.1
> Дата: 8 березня 2026

Цей документ описує стратегії управління великим журналом аудиту в системі Rack Calculator.

---

## 🎯 Проблема

Журнал аудиту може швидко розростатися, особливо в активних системах. Це призводить до:

- Збільшення розміру бази даних
- Уповільнення запитів
- Збільшення часу резервного копіювання

---

## ✅ Рішення

### 1. **Оптимізація запитів (Індекси)**

Міграція `010_add_audit_indexes.js` додає:

```sql
-- Індекс для ефективної пагінації
CREATE INDEX idx_audit_log_created_at_desc 
ON audit_log(created_at DESC);

-- Комбінований індекс для фільтрації
CREATE INDEX idx_audit_log_action_created 
ON audit_log(action, created_at DESC);
```

**Результат:**
- Швидша пагінація на 80-90%
- Ефективна фільтрація за діями

---

### 2. **Автоматичне очищення (Cron)** ⭐

#### Налаштування в .env

```env
# Увімкнути/вимкнути очищення
AUDIT_CLEANUP_ENABLED=true

# Розклад (cron syntax)
AUDIT_CLEANUP_SCHEDULE=0 2 * * 0  # Кожну неділю о 02:00

# Зберігати записи за останні днів
AUDIT_CLEANUP_DAYS=90

# Запускати при старті сервера
AUDIT_CLEANUP_RUN_ON_START=false
```

#### Приклади розкладу

```bash
# Щодня о 02:00
0 2 * * *

# Кожну неділю о 02:00
0 2 * * 0

# Кожного 1-го числа місяця о 03:00
0 3 1 * *

# Кожну понеділок та четвер о 01:00
0 1 * * 1,4
```

---

### 3. **Ручне очищення (CLI)**

```bash
# Видалити записи старіше 90 днів (за замовчуванням)
npm run audit:cleanup

# Видалити записи старіше 30 днів
npm run audit:cleanup 30

# Видалити записи старіше 180 днів
npm run audit:cleanup 180
```

**Особливості:**
- Видалення пакетами по 1000 записів
- Автоматична оптимізація БД (VACUUM) після видалення
- Безпечне видалення без блокування таблиці

---

### 4. **API endpoint**

#### Очищення

```http
POST /api/audit/cleanup
Content-Type: application/json
Authorization: Bearer <token>

{
  "days": 90
}
```

**Відповідь:**
```json
{
  "message": "Deleted 15234 records older than 90 days",
  "deleted": 15234,
  "days": 90
}
```

**Обмеження:**
- Мінімум: 1 день
- Максимум: 365 днів

#### Статистика

### 5. **Інтерфейс адміністратора**

Сторінка `/admin/audit` надає:

#### Статистика
- 📊 **Всього записів** - загальна кількість
- ⏰ **За 7 днів** - активність за тиждень
- 📅 **За 30 днів** - активність за місяць
- 💾 **Розмір БД** - займаний обсяг

#### Фільтрація
- Пошук по тексту
- Фільтр за діями (CREATE, UPDATE, DELETE, LOGIN тощо)
- Фільтр за сутностями (user, price, rack_set тощо)
- Пагінація (50 записів на сторінку)

#### Очищення
- Кнопка "Очистити старі"
- Вибір періоду збереження (1-365 днів)
- Підтвердження операції
- Індикатор виконання

## 🔧 Моніторинг

### SQL запити для аналізу

```sql
-- Загальна кількість записів
SELECT COUNT(*) FROM audit_log;

-- Кількість записів по днях (останні 7 днів)
SELECT DATE(created_at) as date, COUNT(*) as count
FROM audit_log
WHERE created_at >= datetime('now', '-7 days')
GROUP BY DATE(created_at)
ORDER BY date DESC;

-- Топ дій
SELECT action, COUNT(*) as count
FROM audit_log
GROUP BY action
ORDER BY count DESC
LIMIT 10;

-- Розмір таблиці
SELECT page_count * page_size as size_bytes
FROM pragma_page_count(), pragma_page_size();

-- Записи старіше 90 днів
SELECT COUNT(*) FROM audit_log
WHERE created_at < datetime('now', '-90 days');
```

## ⚠️ Важливі зауваження

1. **Не видаляйте занадто часто** - аудит потрібен для відстеження історії
2. **Зберігайте мінімум 30 днів** - для розслідування інцидентів
3. **Робіть бекап перед очищенням** - на випадок помилки
4. **Моніторьте розмір БД** - не допускайте > 500 MB
